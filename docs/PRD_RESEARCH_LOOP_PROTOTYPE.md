# Second Mind 研究闭环原型 PRD

## 1. 产品定位

Second Mind v1 是一个本地优先的个人认知操作系统。当前原型优先验证 Research System，因为研究场景天然具备高噪声、高判断密度和长周期复盘需求。

核心闭环是：

```text
输入 -> 评分 -> 编译 -> 记忆 -> 行动 -> 复盘
```

这个产品不是普通笔记软件，不是通用 RAG 聊天框，也不是图谱可视化工具。它更像用户个人认知流程的“控制台”：帮助用户判断什么重要、应该放在哪里、下一步做什么、什么时候重新检查。

## 2. 验证目标

本原型要验证一个问题：

> 用户是否能够清楚看到并信任一条路径：从零散研究输入，到评分判断，到持久记忆，到下一步行动，再到计划复盘？

成功标准不是自动化深度，而是认知清晰度。用户需要一眼看懂：

- 什么信息进来了
- 为什么它重要
- 它改变了什么判断
- 它应该被编译成哪类记忆
- 下一步行动是什么
- 什么时候复盘

## 3. 目标用户

主要用户是一个研究密集型个人，长期使用 Codex、Obsidian、Zotero 和本地 Markdown 管理技术方向、论文、讨论、决策和行动。

关键用户特征：

- 同时思考研究、投资、生活和 AI 系统
- 更信任本地、可审计、可迁移的记忆层
- 重视判断质量，而不是信息堆积
- 希望系统产出行动和复盘，而不是只生成摘要

## 4. 核心概念

### Source / 输入源

输入源是原始或半原始材料，例如 Zotero 高亮论文、Codex 研究讨论摘要、Obsidian 笔记。

相邻概念是 archive / 归档。归档只是保存材料；Second Mind 要做的是把材料编译成稳定的知识单元。

### Score / 评分

评分是对输入是否值得进入长期记忆的压缩判断。

Research v1 使用的初始评分模型是：

\[
ResearchScore = 0.30R + 0.25E + 0.20N + 0.15A + 0.10U
\]

其中：

- \(R\)：与当前研究方向的相关性
- \(E\)：证据质量
- \(N\)：新颖性
- \(A\)：可行动性
- \(U\)：被低估的潜力

评分的反面是 hoarding / 囤积。囤积保护材料，评分保护注意力。

### Compilation / 编译

编译是把输入材料转化为稳定页面类型的过程，包括 Paper Source、Concept、Direction、Decision、Action。

类比来说，编译不是“把东西丢进文件夹”，而是“把材料整理成图书馆中能被再次使用的书架位置”。

### Memory / 记忆

本地 Markdown 是事实源。Dashboard 只是控制台，不是记忆本体。

这条边界很重要：如果控制台和事实源混在一起，系统会变成一个难以审计的黑箱。

### Review / 复盘

复盘让知识库保持活性。没有复盘层的研究系统，最终会退化为静态仓库。

## 5. 功能范围

### 原型范围内

- Research Loop 控制台
- 三类 v1 输入源列表
- 输入源详情
- 分维度评分解释
- 推荐编译目标
- 持久记忆预览
- 下一步行动队列
- 复盘计划面板
- 跨系统影响提示
- 轻量交互：选择、筛选、编译、查看复盘状态

### 原型范围外

- 真实 Zotero API 接入
- 真实 Obsidian vault 写入
- 真实 Git commit 或同步
- 图谱可视化
- 自动导师日程
- 广泛网页和社交媒体摄取
- 复杂多 Agent 编排
- 生产级认证
- 云端存储

## 6. 用户流程

1. 用户打开 Research Loop 控制台。
2. 用户查看按来源和优先级组织的输入。
3. 用户选择一个输入源。
4. 系统展示评分拆解，并解释为什么它重要。
5. 用户接受或调整推荐编译目标。
6. 系统预览将要形成的记忆条目和下一步行动。
7. 用户将该输入编译到模拟记忆层。
8. 系统显示跨系统影响，并安排复盘。

## 7. 主页面结构

### 控制总览

展示闭环状态、当前阶段、待处理输入、行动、复盘和记忆新鲜度。

### 输入与评分

展示当前输入、来源信息、评分维度和推荐路由。

### 编译预览

展示输出页面类型、摘要、判断影响、下一步行动和复盘日期。

### 复盘队列

展示需要重新检查的内容，包括置信度变化、失效假设和阻塞行动。

## 8. UI 方向

参考方向：Apple-like clarity and restraint，即苹果式清晰、克制和层级感。

设计规则：

- 使用系统字体栈，建立强但不喧闹的文字层级
- 使用安静的中性背景和白色内容面板
- 颜色只用于状态和交互，不做装饰性堆叠
- 卡片圆角控制在 8px 或更低
- 不使用炫技式图谱
- 用可读的操作轨道表达闭环，而不是装饰图
- 主行动清晰，但不使用粗暴的满宽按钮
- 控件保持原生、可访问、可键盘操作

Apple HIG 对本原型的转化：

- Hierarchy / 层级：闭环状态和当前输入优先
- Harmony / 协调：间距、面板、控件几何重复一致
- Consistency / 一致：导航、筛选和状态切换保持稳定
- Accessibility / 可访问性：文字尺寸、对比度、状态标签不只依赖颜色

## 9. 原型数据模型

```text
Source {
  id
  title
  type
  system
  status
  score
  score_dimensions
  recommended_page_type
  why_it_matters

  review_date
  cross_system_links
}
```

## 10. 验收标准

- 用户能从首屏理解完整闭环。
- 选择不同输入后，评分、记忆预览、行动和跨系统影响同步变化。
- 原型让 Research v1 像一个运行中的研究操作系统，而不是论文档案库。
- 本地 Markdown 事实源和 Dashboard 控制台之间的边界清晰。
- UI 保持简洁、干净、克制，有苹果式产品感，但不复制具体苹果页面。

## 11. 用户可能“不知道自己不知道”的风险

- 最大风险不是视觉设计，而是评分规则能否长期产生可信排序。
- 第二个风险是 memory merge / 记忆合并质量：重复概念和过期决策会悄悄污染系统。
- 第三个风险是复盘疲劳：如果太多条目要求复盘，复盘层会变成另一个收件箱。
- 第四个风险是事实源漂移：Obsidian、Codex 摘要、Zotero 元数据和 Dashboard 状态可能彼此不一致。

## 12. 下一步实现切片

静态原型之后，最有价值的下一步是做一个本地文件驱动的 demo：

```text
mock JSON -> dashboard -> generated Markdown preview -> write to local prototype memory folder
```

这能验证 Dashboard 是否真的可以成为“编译器表面”，同时避免过早接入所有真实工具。
